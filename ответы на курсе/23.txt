//Практика 1
public class Calculator {
    // Сложение двух целых
    public int add(int a, int b) {
        return a + b;
    }
    
    // Перегрузка: три целых
    public int add(int a, int b, int c) {
        return a + b + c;
    }
    
    // Перегрузка: два числа с плавающей точкой
    public double add(double a, double b) {
        return a + b;
    }
    
    // Перегрузка: сложение строк
    public String add(String a, String b) {
        return a + b;
    }
}

//Практика 2

class Animal {
    public void sound() {
        System.out.println("Животное издает звук");
    }
}

class Dog extends Animal {
    @Override
    public void sound() {
        System.out.println("Собака лает");
    }
}
Ответ на вопрос:
Если тип возвращаемого значения в подклассе не совпадает с типом в суперклассе и не является его подтипом (ковариантность), то компилятор выдаст ошибку. Например, если в суперклассе метод возвращает int, а в подклассе — String, то это недопустимо, и программа не скомпилируется. Ковариантность разрешена только для ссылочных типов: подкласс может вернуть более конкретный тип (например, String вместо Object), но не наоборот. Для примитивных типов ковариантность невозможна.

//Практика 3 

class Base {
    public void process() { ... }
}

class Derived extends Base {
    @Override
    public void proces() { ... }  // Ошибка: нет метода proces() в Base
}
Аннотация @Override заставляет компилятор проверить, что метод действительно переопределяет метод суперкласса (или реализует абстрактный метод интерфейса). Если такого метода нет, компиляция прерывается. Это помогает предотвратить случайные ошибки, например:

Опечатка в имени метода - нет переопределения, но без @Override компилятор просто создаст новый метод.
Несовпадение сигнатуры (например, другой тип параметра) - без @Override это будет перегрузка, а не переопределение.
Изменение имени метода в суперклассе - подкласс с @Override перестанет компилироваться, напомнив о необходимости синхронизации.
